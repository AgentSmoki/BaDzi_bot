# Промпты для ревью 6 алгоритмов мышления (Wave 7 Phase 4)

> **Цель:** дать сенсею и Богдану структурированные промпты для проверки каждого из 6 драфтов, прежде чем интегрировать их в `base_edoha.md` или skill-каталог.
>
> **Каждый промпт строится по шаблону:**
> 1. Контекст (что это за алгоритм)
> 2. Конкретные точки разбора (5-7 вопросов)
> 3. Запрос на расширение (что добавить, что упростить, что выкинуть)
> 4. Формат ответа (как именно отвечать)
>
> Промпт даётся сенсею ОДНИМ сообщением, отдельно для каждого алгоритма. Ответы сенсея → итерация драфта → итоговый файл готов для интеграции.

---

## Лимит размера алгоритма

| Параметр | Значение |
|---|---|
| Целевой размер | **5-10 KB / 100-200 строк markdown** |
| Минимум | 3 KB — слишком короткий = слишком абстрактный, LLM додумывает |
| Максимум | 12 KB — больше → LLM начинает «забывать» начало алгоритма на длинном ответе |
| Текущий эталон | `risk_assessment.md` = 7 KB / 130 строк ✓ |

В окне 262k tokens Qwen3.6 хватает запаса 230k+, технический потолок ~700 KB. Но **практический потолок** диктуется качеством LLM-следования инструкции, не размером окна.

## 1. opportunity_window.md (благоприятные периоды)

> Текущий драфт: [ai/prompts/algorithms/opportunity_window.md](BaDzi_bot/ai/prompts/algorithms/opportunity_window.md) (~9 KB / 180 строк)

**Промпт сенсею:**

```
Здравствуйте. Прошу ревью алгоритма «opportunity_window» — это chain-of-thought
для AI-консультанта Анастасии, которая будет отвечать на вопросы клиентов про
благоприятные периоды («когда лучше делать X», «когда финансовое окно»).

Ключевая идея алгоритма: благоприятный период = когда в карте уже есть 2 ветви
тройного слияния 三合, а приходящий период приносит 3-ю — образуется полный
треугольник стихии. Это сильнее чем одиночный 桃花 или 貴人.

Прошу разобрать:

1. Корректна ли иерархия силы окон?
   🥇 3-out-of-3 (полная тройка) > 🥈 半合 (полу-слияние) > 🥉 桃花/貴人
   Или эта классификация искажает реальную практику?

2. Нужно ли отдельно учитывать «врождённый ресурс» (когда в карте УЖЕ 3 ветви
   тройки) — драфт говорит «спит без приходящей активации», правильно ли это?
   Если да, что считать «активацией»?

3. 桃花-таблица в драфте даёт правило выбора 桃花 для каждой группы карты
   (申子辰 → 酉, и т.д.). Это базовое классическое правило — но школа ЭдоХа
   как-то его расширяет/уточняет?

4. 天乙貴人 «снижает риски на 50-70%» — это число эмпирическое, или есть
   школьное правило когда вес выше/ниже?

5. Алгоритм не покрывает 化氣格 (трансформации) и 奇門遁甲. Эти границы
   правильные, или нужно затронуть хотя бы базовые трансформации?

6. На эталонной карте Богдана алгоритм находит окно июль 2026 (приходит 未,
   завершает тройку 亥卯未 Дерева). По вашему опыту это действительно
   ключевое окно для него, или есть более значимые периоды которые драфт
   пропускает?

Формат ответа: для каждого пункта 1-3 фразы. Если есть конкретная правка
текста — напишите «было: ... надо: ...».
```

## 2. relationships_match.md (совместимость двух карт)

> Текущий драфт: [ai/prompts/algorithms/relationships_match.md](BaDzi_bot/ai/prompts/algorithms/relationships_match.md) (~10 KB / 200 строк)

**Промпт сенсею:**

```
Здравствуйте. Прошу ревью алгоритма «relationships_match» — совместимость
двух карт по запросу клиента про конкретного партнёра.

Ключевая идея: совместимость читается через 4 уровня резонанса:
(1) столб дня ↔ столб дня, (2) Дворец Супруга — какое 10-е Божество X для Y,
(3) совпадение Юн Шэнь — кто несёт лекарство для кого, (4) кросс-резонансы
остальных столпов.

Прошу разобрать:

1. Правильна ли иерархия уровней? Многие классические разборы дают только
   уровень (1). Драфт добавляет 4 уровня — это избыточно или это и есть
   школа ЭдоХа?

2. Таблица 天干合 интерпретируется через бытовые образы («дуб + почва»).
   Подходят ли эти образы клиенту современного русскоязычного контекста,
   или есть более точные?

3. Дворец Супруга — драфт читает «ветвь дня партнёра Y ↔ ствол дня
   партнёра X = 10-е Божество». Это симметрично (читать в обе стороны)
   или асимметрично (только женщина читает мужа через мужа, не наоборот)?

4. «Юн Шэнь несут друг другу» — это очень благоприятная комбинация.
   Бывают ли карты где это «слишком хорошо» и приводит к зависимости?
   Должен ли алгоритм это отмечать?

5. Алгоритм НЕ работает без столпа дня партнёра (минимум). Многие клиенты
   знают только дату рождения партнёра без времени. Что лучше:
   (а) дать частичный разбор по году+месяцу+дню с предупреждением,
   (б) отказать в разборе пока не дадут точные данные?

6. На эталонной карте Богдана (丁/卯/酉/卯/亥) против гипотетического 壬辰
   алгоритм дал: 天干合 丁壬→Дерево, симметричный 傷官 в Дворце Супруга,
   двойной Юн Шэнь от партнёра. По вашему опыту это «удачная партия»
   или есть скрытые риски в такой комбинации?

Формат ответа: для каждого пункта 1-3 фразы. Если правка — «было: ... надо: ...».
```

## 3. career_alignment.md (карьерное соответствие)

> Текущий драфт: [ai/prompts/algorithms/career_alignment.md](BaDzi_bot/ai/prompts/algorithms/career_alignment.md) (~10 KB / 190 строк)

**Промпт сенсею:**

```
Здравствуйте. Прошу ревью алгоритма «career_alignment» — какая работа
подходит клиенту, когда «окно» для рывка, где деньги пойдут легко.

Ключевая идея: 3 слоя — структура (что не можешь не делать) + талант
(藏干 + 食傷) + окно реализации (когда финансовая стихия активна).

Прошу разобрать:

1. Соответствие «структура → сферы» (таблица в драфте) — это классический
   набор, или школа ЭдоХа что-то меняет? Например, 偏財格 → бизнес, инвестиции,
   риск — но в современной экономике это может быть фриланс, продакт-менеджмент,
   маркетинг. Нужно ли расширить?

2. «Скрытые таланты через 藏干» — драфт говорит «скрытые в дневной ветви
   проявленные личностно, в часовой — для второй половины жизни». Это
   общеклассическое распределение или ваше?

3. 食傷 без 財 = «талант ради таланта, не доход». Это категоричное правило
   или есть исключения (например когда такт удачи Богатства активирует 食傷)?

4. Окно реализации финансов через тройное слияние стихии Богатства — драфт
   даёт пример Богдана: текущий такт 庚午 «смешанный», следующий 辛未 даст
   удвоение Металла + активацию 乙 (Дерево). По вашему опыту так
   действительно происходит — окна совпадают с тактовыми сменами стихий?

5. Алгоритм НЕ покрывает: конкретная зарплата, конкретный город/страна,
   совместимость с конкретной должностью. Эти границы правильные? Что
   ещё стоит явно запретить?

6. Спорный кейс: клиент УЖЕ работает «не в своей сфере» по карте. Что
   лучше сказать — (а) мягко предложить смену, (б) работать с текущим
   контекстом, (в) дать оба варианта?

Формат ответа: для каждого пункта 1-3 фразы. Если правка — «было: ... надо: ...».
```

## 4. decision_chain.md (бинарные решения)

> Текущий драфт: [ai/prompts/algorithms/decision_chain.md](BaDzi_bot/ai/prompts/algorithms/decision_chain.md) (~10 KB / 195 строк)

**Промпт сенсею:**

```
Здравствуйте. Прошу ревью алгоритма «decision_chain» — клиент спрашивает
«делать X или Y?» (новая работа vs остаться, переезд vs нет, проект vs
проект).

Ключевая идея: не давать жёсткий ответ «делай X», а показать «шахматную
позицию» — сила обеих сторон через накопленные ветви, размен (что отдаёт),
3-5 ходов вперёд через ближайшие такты.

Прошу разобрать:

1. Принцип «не давать жёсткий ответ» — это правильное самоограничение,
   или иногда (например 9/10 против 1/10) стоит давать прямой совет?

2. Шахматная аналогия — «жертва пешкой ради ферзя». Подходит ли клиентам
   не знакомым с шахматами? Альтернатива — спортивная аналогия (отдать матч
   ради сезона)? Или нужно подбирать аналогию под контекст клиента?

3. Алгоритм требует чтобы клиент сам описал X и Y в стихиях. Часто
   клиент не понимает «у меня X — это Огонь или Земля». Что лучше:
   (а) задать уточняющие вопросы и ждать ответа,
   (б) самой угадать стихии по описанию,
   (в) дать общий ответ без точной классификации?

4. «Просчитать 3-5 ходов вперёд» — это про ближайшие 5 лет. Что делать
   когда клиент спрашивает про решение на 10-15 лет вперёд?

5. Промежуточный вариант («оставить 20-30% X + 70% Y») — драфт даёт его
   как «жертва пешкой ради ферзя». В вашей практике клиенты выбирают
   гибридный вариант, или они хотят однозначный ответ?

6. Что если клиент уже принял решение и просит подтверждения? Что лучше:
   (а) дать поддержку,
   (б) объективно показать риски выбранного варианта,
   (в) показать оба варианта и оставить выбор?

Формат ответа: для каждого пункта 1-3 фразы. Если правка — «было: ... надо: ...».
```

## 5. health_diagnostics.md (здоровье через баланс стихий)

> Текущий драфт: [ai/prompts/algorithms/health_diagnostics.md](BaDzi_bot/ai/prompts/algorithms/health_diagnostics.md) (~10 KB / 200 строк)

**Промпт сенсею:**

```
Здравствуйте. Прошу ревью алгоритма «health_diagnostics» — диагностика
ТКМ-систем по балансу 5 элементов в карте. Превентивная, НЕ для острых
симптомов.

Ключевая идея: не «слабый элемент = больной орган», а 3 типа дисбаланса:
(1) перенасыщение (40%+ давит контролируемую стихию), (2) пустота (<10%)
+ сильный контролёр, (3) изоляция (стихия есть, но без циклов с другими).

Прошу разобрать:

1. 3 типа дисбаланса — это полная классификация или есть 4-й/5-й?
   Например: «обратный контроль» (Земля слабая, но Дерево очень слабое —
   тогда Земля может «контр-контролировать» Дерево)?

2. Соответствие «стихия → орган» в драфте — это стандартная ТКМ. Школа
   ЭдоХа добавляет что-то к классической схеме (например связь Огня
   не только с сердцем но и с тонким кишечником как путём)?

3. На эталонной карте Богдана драфт даёт топ-3 уязвимых системы:
   ЖКТ+селезёнка (пустота Земли), печень+шея (конфликт перенасыщения
   Дерева и Металла), сердечно-сосуд (изолированный Огонь). По вашему
   разбору карты Богдана — это правильная приоритизация?

4. Сезонные пики (март/июнь/сентябрь) — драфт даёт их автоматически по
   приходящим столпам. Но в реальности у клиента сезонные симптомы могут
   сдвигаться (например пик весны в феврале или мае). Что лучше:
   (а) держать жёсткую привязку к месяцам по 节气,
   (б) допускать диапазоны,
   (в) корректировать по обратной связи клиента?

5. Запреты — драфт обязывает Анастасию НЕ давать диагнозы и НЕ предсказывать
   психиатрию. Что ещё в категории «нельзя» в вопросах здоровья?

6. ТКМ-диетология (конкретные продукты) — это в компетенции Анастасии или
   нужно мягко переключать на профильного специалиста?

Формат ответа: для каждого пункта 1-3 фразы. Если правка — «было: ... надо: ...».
```

## 6. term_unpack.md (мета-алгоритм расшифровки терминов)

> Текущий драфт: [ai/prompts/algorithms/term_unpack.md](BaDzi_bot/ai/prompts/algorithms/term_unpack.md) (~9 KB / 180 строк)

**Промпт сенсею (мета-алгоритм — отдельная природа):**

```
Здравствуйте. Прошу ревью алгоритма «term_unpack» — это НЕ алгоритм разбора
карты, а МЕТА-правило: как Анастасия расшифровывает китайские иероглифы и
классические термины Ба Цзы для клиента.

Контекст: после Phase 1 (commit 0ad9d29) мы ввели правило «голый иероглиф =
критическая ошибка». Сейчас в каждом ответе при первом упоминании 七殺 идёт
расшифровка «Семь Убийств — энергия давления как строгий начальник». Этот
алгоритм формализует правило.

Прошу разобрать:

1. Категории терминов требующих расшифровки (драфт: иероглифы, классические
   русские термины, школьный жаргон, неповседневная лексика) — полная?

2. Шаблон расшифровки — 3 уровня глубины (минимальная / средняя / подробная).
   Когда какой использовать? Драфт говорит «по умолчанию минимальная».
   Не упрощает ли это до банальности?

3. Правила повторных упоминаний — «один раз в сообщении, заново в каждом
   новом ответе». Это раздражает клиента (повторение) или защищает (он мог
   забыть)? Что предпочесть?

4. Запрет на сокращения «ДМ», «ЮнШэнь» — это категорично или есть случаи
   когда сокращение уместно (например в техническом разборе для продвинутого
   клиента)?

5. «Не объяснять термин через другой нерасшифрованный термин» — например,
   нельзя «Юн Шэнь — это полезное Божество». Как обходить когда термины
   связаны цепочкой?

6. Алгоритм предлагает программную пост-проверку (regex на иероглифы без
   скобок). Это перебор или нужно реализовать как safety net?

Формат ответа: для каждого пункта 1-3 фразы. Если правка — «было: ... надо: ...».
```

## После сбора ответов

1. Каждый драфт правится по фидбеку (1 коммит на драфт).
2. Решается интеграция в runtime (Опция A/B/C из `doc/school_layered_flow.md`):
   - **A** — каждый алгоритм отдельным skill (расширение `SkillName` Literal)
   - **B** — все embedded в `base_edoha.md` (раздутый prompt, но всё доступно)
   - **C** — conditional injection через mapping (skill + school → algorithm)
3. Тесты intent → алгоритм мэппинга через MCP live-verify.
