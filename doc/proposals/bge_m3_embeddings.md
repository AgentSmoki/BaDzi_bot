# Предложение: семантический RAG через bge-m3 эмбеддинги

> **Статус:** предложение, к запуску по решению Богдана (НЕ внедрено).
> **Дата:** 2026-06-02. **Контекст:** Wave 7, после импорта 7742 узлов ЭдоХи.
> **Решение по инфраструктуре:** остаёмся на **KuzuDB 0.10** (Apache AGE отменён) — bge-m3 этому не мешает.

## Проблема

Текущий retrieval ([ai/rag/retrieve.py](../../ai/rag/retrieve.py)) — **sparse** (по словам):
- overlap по `related_concepts`,
- русский стемминг токенов вопроса,
- `lower(n.title) CONTAINS`.

На 7742 узлах ЭдоХи это молчит, когда в вопросе **нет точного слова** концепта. Пример: вопрос «как мне перестать выгорать на работе» лексически не пересекается с цитатой мастера про «истощение Огня» / «спринтер на марафоне» — релевантный узел не находится. Recall падает тем сильнее, чем больше корпус.

## Что предлагается

Добавить **dense**-поиск по смыслу поверх существующего sparse (гибрид):

1. **Модель — bge-m3** (BAAI): 568M параметров, 1024-мерные векторы, мультиязычная (рус + кит нативно), лицензия MIT. Лучшее покрытие для нашего корпуса (≈70% русский + 30% китайская терминология) в своём весовом классе.
2. **Indexing (один проход на ingest):** прогнать `body`/`summary` каждого узла через bge-m3 → записать вектор в уже существующую колонку `embedding FLOAT[]` в [knowledge/schema.py](../../knowledge/schema.py) (сейчас пустая, зарезервирована под Phase 2.5).
3. **Query:** эмбеддинг вопроса → косинусная близость против узлов → top-k.
   - **На KuzuDB 0.10 без Apache AGE:** косинус считаем в Python над школо-отфильтрованной выборкой кандидатов (school-clause из retrieve.py остаётся), либо нативной array-функцией расстояния Kuzu, если доступна в 0.10.
   - **Гибрид:** dense-score + текущий sparse-score → re-rank top-k. Чистый dense иногда тащит «похоже по смыслу, но не то»; sparse держит точные термины (七殺, 桃花). Вместе — лучше каждого по отдельности.

## Размещение модели (две опции)

| Опция | Плюсы | Минусы |
|---|---|---|
| **A. Локально в worker** (sentence-transformers) | Без внешних зависимостей, приватно | +~500 МБ к Docker-образу, +RAM на инференс |
| **B. По API** (эмбеддер как сервис) | Образ не растёт | Сетевая зависимость, биллинг за эмбеддинги |

Рекомендация: начать с **A** (один прогон на ingest + лёгкий инференс запроса), при росте нагрузки — вынести в отдельный сервис.

## Объём работ (оценка)

- `knowledge/ingest/embed.py` — батч-эмбеддинг всех узлов + запись в `embedding`. (~0.5 дня)
- `ai/rag/retrieve.py` — dense-кандидаты + гибрид re-rank, school-clause сохранить. (~1 день)
- Перепрогон ingest на 7788 узлах (один раз) + `docker compose cp` kuzu_db на VM.
- Тесты: cosine ranking, гибрид-merge, fallback на sparse при пустых embedding. (~0.5 дня)

Итого ~2 дня. **Триггер уже выполнен** (корпус >100 docs, 7788 узлов) — упирается только в решение о запуске.

## Что НЕ входит

- Векторный индекс/HNSW (для 7788 узлов линейный косинус по кандидатам достаточен).
- Миграция на Apache AGE / pgvector — **отменена**, остаёмся на KuzuDB 0.10.
- Дообучение модели — берём bge-m3 как есть.
