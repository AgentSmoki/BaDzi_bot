Две доработки по фидбэку с прода:

- **Задача 1.** Приглашение записать рефлексию о важной дате приходит в **12:00** (глобальный скан `scan_important_dates_job` в 09:00 UTC = 12:00 MSK). Нужно слать **вечером ~18:00 по местному времени** карты — день прожит, есть что рефлексировать.

## Задача 1 — рефлексия о важной дате в 18:00 местного времени

**Сейчас:** [scan_important_dates_job](bot/scheduler/jobs.py) (cron 09:00 UTC) шлёт и предупреждение (за 1-2 дня), и day-of рефлексию — обе в 12:00 MSK.

**Решение:** разнести по времени.
- **Предупреждение** остаётся в глобальном скане 09:00 UTC (за 1-2 дня, утром — ок).
- **Рефлексия (day-of)** переезжает в новый **почасовой** глобальный джоб, который шлёт её в 18:00 локального времени карты.

Почему почасовой глобальный, а не per-chart cron: `important_dates_enabled=True` по умолчанию у ВСЕХ карт → per-chart регистрация дала бы тысячи джобов. Почасовой скан с фильтром `reflection_hour_utc == текущий_час` дёшев и масштабируется.

**Изменения:**

1. **`db/models.py` `ChartJournalSettings`:** добавить `reflection_hour_utc: Mapped[int]` (default 15 = 18:00 при UTC+3). + миграция: `add_column` + backfill из `charts.tz_offset`:
   `UPDATE chart_journal_settings SET reflection_hour_utc = ((18 - round(c.tz_offset))::int % 24 + 24) % 24 FROM charts c WHERE chart_journal_settings.chart_id = c.id`.
2. **`db/repositories/journal_repo.py`:**
   - `list_reflection_due_at(session, hour_utc)` → `important_dates_enabled=True AND reflection_hour_utc == hour_utc`.
   - В `toggle_important_dates(...)` опциональный `tz_offset: float | None`; если задан — `reflection_hour_utc = _hour_local_to_utc(18, tz_offset)`. Прокинуть `chart.tz_offset` из вызовов в [start.py](bot/routers/start.py) (chart-меню) и [forecast.py](bot/routers/forecast.py) (покупка прогноза).
3. **`bot/scheduler/jobs.py`:**
   - Новый `scan_reflection_prompts_job()`: `hour = datetime.now(UTC).hour`; `list_reflection_due_at(hour)`; для каждой карты `find_important_dates_in_range(chart, today, today)`; если hit и `last_reflection_prompt_date != today` → `format_important_date_reflection` + кнопка `journal:write` + `mark_reflection_prompt_sent` + commit per-chart.
   - Из `scan_important_dates_job` **удалить Block 1** (day-of рефлексия), оставить Block 2 (warning). `_hour_local_to_utc` уже в jobs.py.
4. **`bot/scheduler/runner.py`:** зарегистрировать `scan_reflection_prompts_job` глобально `CronTrigger(minute=0)` (каждый час), id `reflection:hourly_scan`, `replace_existing=True`, `misfire_grace_time=3600`.

**Тесты:** `test_scan_important_dates.py` — `scan_important_dates_job` шлёт только warning; новый `scan_reflection_prompts_job` шлёт рефлексию при `reflection_hour_utc==now.hour` + сегодня важная дата, и НЕ шлёт при несовпадении часа / уже отправленной сегодня.

## Задача 2 — возрастные метафоры по школам (все 3 школы)

**Принцип:** фича работает во ВСЕХ школах, но **метафоры у каждой школы свои** и **под каждый возрастной бэнд свои**. Контекст метафоры — **любое пересечение звёзд** в карте (столкновение 六冲, наказание 刑, вред 害, слияние 合/三合, активация 桃花/天乙貴人/将星 и т.п., встреча 10 богов), не только столкновения.

### 2.1 Возраст → в промпт
- **Утилита** `calculator/age_utils.py` (new): `client_age_years(birth: datetime) -> int` (по календарю, не `//365`). Возраст доступен через `ChartOutput.input.birth_datetime` ([calculator/models.py](calculator/models.py)).
- **`ai/temporal_context.py::compose_messages`** — параметр `client_age: int | None = None`; после `[BAZI_DATA]` инжектить нейтральную секцию:
  ```
  [AUDIENCE]
  Возраст клиента: <N> лет (бэнд <label>).
  При расшифровке звёзд и пересечений в карте подбирай метафоры под этот возраст и в стиле своей школы (см. «Возрастные метафоры» в системном промпте).
  [/AUDIENCE]
  ```
  Бэнд-лейбл по N — короткая мапа в коде (бэнды: `<25`, `25-35`, `35-45`, `45-60`, `60+`).
- **`bot/routers/consultation.py::_continue_consultation_with_skill`** — вычислить `client_age` из `chart_data.input.birth_datetime`, передать в `compose_messages`. (Там же, где `school=chosen_school`.)

### 2.2 Промпт подбора метафор (ОБЩИЙ — в `ai/prompts/base.md`)
Требование действует для всех школ → в `base.md` (грузится под каждой школой). Добавить раздел:

```
## Возрастные метафоры (обязательно при расшифровке)

Когда расшифровываешь звезду (Шэнь Ша) или любое пересечение в карте
(столкновение 六冲, наказание 刑, вред 害, слияние 合/三合, активацию
桃花 / 天乙貴人 / 将星, встречу 10 богов) — НЕ оставляй сухой термин.
Дай 2-3 конкретные метафоры-примера, КАЖДУЮ отдельной строкой-цитатой:
> «Представь, что …»

Как подбирать:
1. ВОЗРАСТ — из секции [AUDIENCE]. Бери образы из мира, понятного
   этому возрасту.
2. СТИЛЬ — по своей школе (см. свою базу «Возрастные метафоры» в
   школьном блоке ниже): классика — традиционные/природные образы,
   ЭдоХа — поле/война/шахматы/спорт/животные, современная —
   психология/архетипы/современная жизнь.
3. КОНТЕКСТ — под конкретное пересечение: что с чем сталкивается/
   сливается и в какой сфере жизни клиента это проявляется.
4. Минимум 2-3 метафоры. Пусть ответ будет длиннее — зато нагляднее.
```

### 2.3 Базы примеров — В КАЖДОМ `base_<school>.md` своя
В каждый из [base_classic.md](ai/prompts/base_classic.md), [base_edoha.md](ai/prompts/base_edoha.md), [base_modern.md](ai/prompts/base_modern.md) добавить раздел «## Возрастные метафоры (база примеров)» — таблица бэнд → темы/образы в стиле школы + 1-2 готовых примера. ЭдоХа переиспользует существующие «8 семей метафор» (§5) и 六冲-сцены, привязав их к бэндам.

**Скелет баз (бэнды × школа):**

| Бэнд | Классика (традиц./природа) | ЭдоХа (поле/спорт/животные) | Современная (психология) |
|---|---|---|---|
| <25 | ученик и наставник, экзамен, посев | спарринг, турнир, гейминг, первая любовь | самооценка, дружба, соцсети |
| 25-35 | служба, контракт, лестница чинов | стартап, переговоры, спорт-достижения | карьерный рост, выгорание, границы |
| 35-45 | управление домом/делом, урожай | шахматы, команда, родительство | баланс, кризис среднего, родительство |
| 45-60 | мастер передаёт ремесло, сад | наставничество, второе дыхание, природа | переосмысление, психосоматика |
| 60+ | времена года, корни рода | сад, традиции, внуки | принятие, наследие, мудрость |

**Пример одного пересечения (卯酉冲) на бэнд 30-45 в трёх школах:**
- Классика: `> «Петух — Металл, что подрезает Дерево-Кролика: порядок обрезает лишние побеги. Больно ветке, но дерево крепнет.»`
- ЭдоХа: `> «Два кота-кролика наседают на петуха: гибкость против дисциплины — тянет в разные стороны.»`
- Современная: `> «Внутри спорят двое: спонтанный творец (Кролик) и внутренний контролёр (Петух) — им нужно научиться договариваться.»`

# Следующие задачи в работу (поставлены Богданом 2026-06-10)

## A. Semantic embeddings для RAG (1.9.16) — одобрено

План: [doc/proposals/bge_m3_embeddings_plan.md](doc/proposals/bge_m3_embeddings_plan.md). Оценка ~1.5 дня.

Ключевое решение (live-probe 2026-06-10): **гоним через Yandex Cloud Embeddings API**, не через макбук и не через модель в Docker:
- модели `text-search-doc` (индексация) + `text-search-query` (вопрос), 256-dim, работают с текущим ключом `YC_AI_API_KEY` — проверено живым вызовом;
- индексация 7788 узлов KuzuDB = HTTP-вызовы (~15-30 мин), query-эмбеддинг ~100-300 мс с Redis-кэшем;
- ноль RAM на VM, образ не растёт; fallback на текущий sparse сохраняется;
- quality-gate на эталонных вопросах; если 256-dim модель провалит китайскую терминологию — Вариант 2: bge-m3 ONNX int8 (~700 MB, влезает в VM).

Порядок: Phase 0 (`ai/embeddings.py` + Settings) → Phase 1 (`knowledge/ingest/embed.py`) → Phase 2 (гибрид в `ai/rag/retrieve.py`) → Phase 3 (quality-gate + деплой) → Phase 4 (тесты).
